# Preproduction Brief — The Search Snippet Is Not the Page

## Working title
**The Search Snippet Is Not the Page**

## One-sentence promise
A search result can quote something real and still mislead you, because the preview layer is not the underlying page and may lag, truncate, or freeze an older framing.

## Why this fits Verify the Rails
This concept stays squarely inside the channel’s lane:
- conflicting claims that are both rooted in real artifacts
- evidence hierarchy made visible
- practical verification steps for non-experts
- calm explanation instead of dunking on the person who shared the wrong impression

It also has one strategic advantage over `The Screenshot With No Date`:
- the main lesson may be easier to stage through **layer differences** (snippet vs page) rather than relying as heavily on tiny supporting microcopy.

## Core viewer question
**“Why does the search result say one thing when the page itself now seems to say another?”**

## Candidate disagreement patterns

### Pattern A — Snippet froze an older wording
- Search result preview shows a sentence that was true earlier.
- Live page has since changed or added qualifiers.
- Repost treats the snippet as if it were the current page itself.

### Pattern B — Snippet truncation changed emphasis
- Search result shows the sharpest phrase.
- The missing clause on the live page narrows or qualifies the claim.
- Viewer error comes from trusting the preview layer as a complete quote.

### Pattern C — Social/embed preview vs page content
- Shared card or preview text persists after the underlying article/help page changes.
- People argue from different layers without realizing they are checking different surfaces.

## Target viewer takeaway
Before you argue from a preview, open the page itself.

Secondary habit:
- check whether the preview is quoting an old version, a truncated sentence, or a generated summary layer

## Visual promise
The piece should let the viewer *feel* the mistake:
1. start on the snippet / preview layer because it looks authoritative
2. open the page and reveal the missing context
3. mark the difference between:
   - preview text
   - live page text
   - date / update clues
4. end with a short checklist about preview-vs-source verification

## Possible opening line
**“This preview was real. The page was real. The mistake was treating the preview as the page.”**

## Possible closing line
**“A preview can point you to a source. It is not the source. Open the page before you share the claim.”**

## Evidence requirements
Any eventual production version should include:
- at least one synthetic example with traceable timeline logic
- an explicit visual hierarchy between preview layer and source page
- a clear check timestamp / update clue
- a reusable viewer action that does not depend on technical jargon

## Main risks
1. **Too similar to generic ‘read the article’ advice**
   - fix by focusing on layer mismatch, not scolding
2. **Could become brand-specific or dated**
   - fix with synthetic but realistic UI treatment
3. **Could drift into abstract UX commentary**
   - fix by keeping one concrete disagreement chain central

## Current judgment
This is a **promising backup concept** because it looks capable of carrying its meaning through visual structure rather than tiny explanatory text.

It is not yet outlined, scripted, storyboarded, or greenlit.
